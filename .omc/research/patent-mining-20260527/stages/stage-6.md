# Stage-6: UART通信协议专利挖掘报告

## 总体判断

`uart_feature_streamer.v` 实现了一个**专为传感器物理身份认证设计的UART输出协议**，核心特征：

1. **多帧类型顺序输出**：STATUS → STABLE → RAW → PEAKS → SPECTRUM → DEBUG → STATE_LOG
2. **ON/OFF双态标记**：每帧头部根据 `is_power_off` 自动切换前缀（如 `SPECTRUM_CH1` / `OFF_SPECTRUM_CH1`）
3. **ASCII/二进制双模式**：`mode_binary=0` 时输出人类可读ASCII文本；`mode_binary=1` 时输出带Magic字节的二进制帧
4. **可裁剪输出模式**：`mode_fast_scan`（仅频谱）、`mode_raw_data`（是否含RAW）、`mode_full_spectrum`（是否含128bin全谱）
5. **状态机驱动字符级格式化**：用单状态机逐字符生成格式化文本，而非先缓存再发送

**关键问题**：这种"身份认证专用的UART协议设计"是否可专利？

结论：**不建议作为独立专利申请**，但存在**两个有价值的从属专利点**，可作为主案（01/02/03）的从属权利要求或技术秘密保护。

---

### 专利点1: 上下电双态标记的多帧类型UART输出协议

- **一句话描述**: 在单条UART输出流中，通过帧头前缀自动区分上电响应帧和下电响应帧，按固定顺序输出STATUS、STABLE、RAW、PEAKS、SPECTRUM、DEBUG、STATE_LOG七类数据帧，使认证端能够一次性接收完整的传感器瞬态身份响应包。

- **技术效果**:
  - 上电/下电双态数据在同一串口流中不混淆，无需额外控制线或握手协议区分ON/OFF
  - 七类帧覆盖从原始时域数据到频域峰值到调试信息的完整观测链，支持多级认证策略（快速峰值比对→全谱比对→原始波形复核）
  - 固定顺序输出使PC端解析器无需状态机，仅用行首关键字即可分流处理
  - 帧头前缀由 `is_power_off` 信号实时切换（如 `PEAKS_CH1_B` → `OFF_PEAKS_CH1_B`），RTL实现零额外存储开销

- **证据文件**:
  - `rtl/uart_feature_streamer.v` 第3-13行（协议格式注释）、第276-335行（STATUS帧）、第337-405行（STABLE帧）、第407-497行（PEAKS_CH1帧，含ON/OFF分支）、第591-697行（SPECTRUM帧，含ON/OFF分支）
  - 第89-105行（状态机状态定义，13个状态覆盖全部帧类型）
  - 第988-1006行（STATE_LOG帧，10字节固定格式 `ST,X,Y,ZZ\n`）

- **与现有三件专利的关系**:
  - **01_上下电瞬态响应传感器身份提取**：本专利点可作为其从属权利要求，具体说明"数据输出模块"如何通过UART协议将ON/OFF双态响应封装为顺序帧流
  - **02_SCALE_SCH挑战扫描响应图谱生成**：STATUS帧中携带 `SCALE_SCH index` 和 `SCALE_SCH value`（第320-327行），直接服务于挑战-响应协议的参数传递
  - **03_多特征融合稳定比特量化**：DEBUG_BINS帧（第699-821行）输出峰值bin索引，可用于验证bit化前的峰值定位一致性

- **需要补的实验**:
  - 测量完整帧流输出时间（ASCII模式 vs 二进制模式），证明在认证场景中的实时性
  - 验证PC端解析器对ON/OFF混合流的正确分流率（1000次以上采集无错帧）
  - 证明固定七帧顺序相比自定义/可变顺序在工程实现上的可靠性优势

- **新颖性风险**:
  - **中高**。通用UART多帧协议（如Modbus、NMEA）已有大量现有技术。本方案的新颖性仅在于"上下电双态标记+身份认证专用七帧结构"的组合，单独主张UART协议本身容易被现有技术覆盖。
  - **规避建议**：不作为独立申请，而是作为01/02主案的从属权利要求，绑定"传感器瞬态身份认证"这一特定应用场景。

### 专利点2: 身份认证数据流的ASCII/二进制双模式自适应切换机制

- **一句话描述**: 同一传感器身份认证系统支持ASCII文本模式（人类可读、调试友好）和二进制模式（带宽高效、带Magic字节和事务ID framing）的实时切换，通过单比特控制信号 `mode_binary` 在发送首帧前决定整包输出格式。

- **技术效果**:
  - ASCII模式（`mode_binary=0`）：输出 `STATUS,MS,P,EC,SS,V5.6,F,III,SSS\n` 等人类可读文本，便于调试和日志记录；频谱数据格式化为5位十进制数字，逗号分隔
  - 二进制模式（`mode_binary=1`）：输出带 `0xA5 0x5A` Magic字节、协议版本号、帧类型ID、事务ID（`txn_id`）、负载长度的结构化二进制帧；频谱数据以原始32位小端字节流输出，带宽提升约5-8倍
  - 双模式共享同一状态机骨架，仅首帧分支不同（第264-273行），RTL资源开销极小
  - 二进制帧头包含 `is_power_off` 标志（第1018行/第1056行），保留ON/OFF区分能力

- **证据文件**:
  - `rtl/uart_feature_streamer.v` 第30行（`mode_binary` 参数定义）、第264-273行（IDLE状态双模式分支）
  - 第146-152行（二进制常量定义：Magic字节 `0xA5 0x5A`、协议版本 `0x01`、帧类型 `0x11/0x12`）
  - 第1008-1081行（二进制频谱帧发送逻辑，CH1/CH2各14字节头+512字节负载）
  - 第31行（`txn_id` 事务ID输入，用于二进制帧的帧级关联）

- **与现有三件专利的关系**:
  - **01_上下电瞬态响应传感器身份提取**：可作为系统权利要求中"数据输出模块"的具体实施方式，强调调试阶段用ASCII、量产认证用二进制的工程灵活性
  - **02_SCALE_SCH挑战扫描响应图谱生成**：二进制模式的事务ID（`txn_id`）可与挑战参数索引关联，支持多挑战批量采集时的帧级对齐
  - **03_多特征融合稳定比特量化**：二进制模式减少传输带宽，使高频率重复采集（用于稳定bit筛选）在物理上可行

- **需要补的实验**:
  - 量化ASCII模式 vs 二进制模式的传输时间比（理论：ASCII频谱帧781字节 vs 二进制526字节，提升约48%）
  - 验证二进制帧在误码/丢包场景下的鲁棒性（Magic字节同步恢复机制）
  - 测量 `mode_binary` 切换对FPGA资源（LUT/FF）的影响

- **新颖性风险**:
  - **中**。通信协议的双模式（文本/二进制）设计在通用领域已有先例（如HTTP/HTTPS、某些传感器协议）。新颖性在于"身份认证场景下的自适应切换+Magic字节framing+事务ID关联"的组合。
  - **规避建议**：作为系统专利的从属权利要求，强调"在资源受限FPGA上实现的双模式身份数据流输出"，绑定硬件约束和应用场景。

### 专利点3: 基于字符计数器的状态机驱动格式化发送方法（技术秘密/从属点）

- **一句话描述**: 在FPGA中不借助外部存储器或帧缓存，仅用一个16位字符计数器驱动case语句状态机，实时将多源数据（峰值、频谱、原始采样）格式化为UART字符流逐字节输出。

- **技术效果**:
  - 零BRAM开销：不需要预先组装整行数据到缓冲区，每个字符由计数器位置实时计算（如 `get_digit_16(value, pos)` 函数）
  - 频谱数据通过 `bin_pos = (char_counter - 13) / 6` 和 `digit_in_bin = (char_counter - 13) % 6` 实时定位（第139-140行），无需存储中间结果
  - 256点RAW数据通过7字符循环周期（逗号+符号+5位数字）实时格式化（第823-907行），支持分页输出
  - 状态转移由 `line_type` 和 `char_counter` 阈值共同决定（第1091-1308行），形成自包含的格式化流水线

- **证据文件**:
  - `rtl/uart_feature_streamer.v` 第119行（`char_counter` 寄存器）、第139-140行（频谱bin位置计算）
  - 第173-217行（`hex_digit`、`get_digit_16`、`signed5_char`、`freq_khz_char` 格式化函数）
  - 第276-335行（STATUS帧逐字符case）、第591-643行（SPECTRUM_CH1帧逐字符case）
  - 第1091-1308行（ST_WAIT_TX_DONE状态的完整帧转移逻辑）

- **与现有三件专利的关系**:
  - 这是**底层实现细节**，更适合作为**技术秘密**保护，或在主案系统权利要求中作为"数据输出模块"的一种具体实施方式。
  - 若主张专利，需强调"在资源极度受限的FPGA边缘设备上，无缓存实时格式化多类型身份数据"的技术效果。

- **需要补的实验**:
  - 资源对比：本方案 vs "先缓存整帧再发送"方案的BRAM/LUT消耗对比
  - 时序验证：最高时钟频率下字符计数器递增和case语句是否满足时序约束

- **新颖性风险**:
  - **高**。状态机驱动字符输出是FPGA UART设计的常规做法，单独主张创造性不足。
  - **建议**：不单独申请，作为技术秘密或主案从属权利要求的实施例。
