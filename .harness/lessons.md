# 硬性约束规则（最高优先级）

以下规则被用户纠正后确立，必须严格执行：

| # | 规则 | 违反后果 |
|:---|:---|:---|
| 1 | **脚本 bug 当场修**：任何 harness 脚本 exit code ≠ 0 必须定位并修复，不允许以"核心成功"为由放过 | 累积技术债务 |
| 2 | **长耗时/阻塞任务必须后台执行**：build/program/capture 等 >30s 任务，一律委托子 agent 后台执行 | 阻塞等待浪费用户时间 |
| 3 | **等待期间并行干活**：后台任务运行期间，必须开子 agent 或自己做独立的代码分析/文档/修复 | 单线程效率低下 |
| 4 | **大文件/跨模块分析必须派子 agent**：>400 行 RTL 或跨多文件分析，必须派子 agent 先行扫描 | 注意力分散漏关键 bug |
| 5 | **先读 STATUS 再动手**：看到数据异常，先读 UART STATUS 定位，不允许凭假设改代码 | 盲目修复导致回退 |
| 6 | **用户授权 ≠ 跳过验证**：用户说继续后用省下的时间做更深验证，不是跳过 | 烧录无效修复 |
| 7 | **build 前必须 check 通过**：每次 build 前确保 lint/sim 全绿，禁止盲目 build | 浪费时间 |
| 8 | **禁止瞎猜，必须获取具体信息**：看到错误，第一反应是加调试获取具体变量值，不允许在脑中罗列假设 | 永远猜不到真相 |
| 9 | **子代理服务禁止阻塞**：子 agent 返回结果后立即继续；结果不足时明确问用户或派新子 agent | 浪费令牌和时间 |
| 10 | **Harness 优先原则**：发现 harness 有问题，暂停 RTL 编码先修 harness | 累积技术债务 |
| 11 | **勿增实体原则**：每次编码只推进一个功能，完成后再开下一个。禁止同时改多个不相关模块 | scope 蔓延 |
| 12 | **写完代码必须出具 QA 报告**：任何 RTL/脚本修改后，向用户呈递 QA 报告。禁止只发"改完了" | 用户无法判断改动安全性 |
| 13 | **独立验证零容忍**：QA 报告必须包含真实执行的 check/lint/sim 输出，不允许用"逻辑推演"代替 | 把假设当事实 |
| 14 | **bug 修复后必须写行内注释**：在出 bug 的代码旁注释记录：问题/根因/修复/验证命令。同时在 lessons.md 新增 | 下次同坑从零调试 |
| 15 | **修脚本后必须当场验证**：修改 .ps1/.tcl 后，先用最小命令验证无报错再启动完整流程 | 同坑反复踩 |
| 16 | **工具路径必须硬编码**：Vivado/Verilator/Python 路径硬编码，禁止依赖 PATH 或动态检测 | PATH 不可靠 |
| 17 | **长任务必须主动监控和汇报**：>30s 任务启动后设 monitor 或定时检查进度；关键节点主动汇报 | 用户被动等待 |
| 18 | **被纠正后必须写 lessons.md**：每次被纠正立即追加三要素（问题/根因/修复）；2 分钟内不写视为违规 | 口头认错不沉淀 |
| 19 | **用户说多开就多开**：不自行设 agent 数量上限。文件写入类任务串行防冲突，纯研究/搜索类按用户指示并行 | 系统拦截 |
| 20 | **模块间连接必须用 wire** — 顶层连接处必须是 wire，不能用 reg | 综合警告、时序异常 |
| 21 | **组合逻辑优先用 wire + assign** — 避免 always @(*) case 内跨分支赋值 | 综合结果不确定 |
| 22 | **位宽字面量不得截断** — 2048 需 12'd2048；17 需 5'd17 | 隐式截断导致算术错误 |
| 23 | **只使用 Verilog-2001** — 禁用 SystemVerilog 语法（logic、always_comb、$WIDTH'(expr)） | Verilator lint 失败 |
| 24 | **RTL 改动必须先列改动点等用户确认** — 任何 RTL 改动 → 列具体改动点 → 等用户说"好/执行/改" → 再动手。不推，不猜，不趁热 | 未经授权的改动 |
| 25 | **新功能/实验性改动必须开新 git 分支** — 分支命名 feature/<描述> 或 experiment/<描述> | 回退困难 |
| 26 | **模块间握手必须显式两阶段确认** — 发方设 req=1 → 等收方 ack=1 → 撤 req=0 → 等 ack=0 → 才发下一个。禁止 if (!busy) 单条件直接下一拍 | pipeline 竞态丢数据 |
| 27 | **RTL 改动必须 TDD** — 先写 testbench 覆盖要改的逻辑 → 跑 sim 看它挂 → 改 RTL → 跑 sim 看它过 → 然后才 lint/build/program。禁止 sim 过了就以为板子也能过 | 未验证的 RTL 上板可能死锁/静默 |

## 2026-06-05 V6.0 capture_uart_streamer pipeline 竞态 + 最小化调试法

**问题**：V6.0 完整固件 UART 输出帧头乱码、ADC 值 4 位 hex 变 2 位、TXN 递增异常。

**根因**：`capture_uart_streamer.v` 用 `if (!tx_busy)` 作为发下一字节的条件，但 `tx_busy` 从 `uart_tx` 通过非阻塞赋值传回，有 1 周期 pipeline 延迟：
- 周期 N：streamer 设 tx_start=1，uart_tx 的 busy 还没变
- 周期 N+1：streamer 读 tx_busy 还是 0，误判"空闲"又发一次
- → 每隔一个字节丢一个

**修复**：重写握手为三阶段——发 tx_start=1 → ST_WAIT_ACK（等 tx_busy=1 确认收到）→ 回到发送状态（等 tx_busy=0 确认完成）→ 发下一个。

**涉及文件**：`rtl/capture_uart_streamer.v`（完全重写状态机）

**教训 — 最小化隔离调试法**：
1. 裸 uart_tx 测试（只 3 个模块）→ 验证 UART 物理层正确
2. uart_tx + capture_uart_streamer（固定 send 发一帧）→ 定位 bug 在 streamer
3. 修复后合入完整固件 → 端到端验证 → 避免在 10+ 模块的完整设计中盲调

## OMC 子代理调度表（硬性约束）

**最低强制调度**（缺一即视为违反规则 9/13）：
- **编码 → QA**：写代码 → 审查 → 独立验证 → 汇总 QA 报告
- **诊断 bug**：debugger + tracer 并行 → 主 agent 综合根因
- **新功能设计**：planner + architect → 出 plan
- **依赖工具链**：先查官方文档，禁止凭记忆写
